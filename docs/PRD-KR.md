## 제품 비전 요약

WriteAlive는 작가들이 "살리고 글쓰기"를 실천할 수 있도록 돕는 AI 기반 글쓰기 도구입니다. **김창준**이 개발한 이 방법론은 Christopher Alexander의 "The Nature of Order"와 Bill Evans의 단계별 숙달 철학에서 영감을 받았습니다. 전통적인 개요 우선 접근 방식을 강요하는 대신, WriteAlive는 낮은 에너지 장벽으로 씨앗 중심에서 아이디어가 유기적으로 성장하는 반복적이고 생성적인 글쓰기 프로세스를 지원합니다. 이 도구는 글쓰기를 고마찰, 구조 중심 활동에서 유동적이고 발견 주도적인 창작 과정으로 전환합니다.

**핵심 철학**: 글쓰기는 경직된 계획을 통해 생산되는 정적 산물이 아니라, 생성적 순서를 통해 진화하는 살아있는 구조입니다.

**주요 가치 제안**: 작게 시작하고, 진실되게 구축하며, 자연스럽게 진화함으로써 누구나 명확성과 깊이 있는 글을 쓸 수 있도록 지원합니다 - Bill Evans가 음악가들에게 한 번에 한 단계씩 재즈를 마스터하도록 가르친 것처럼. 이제 탐색적 발견(사용자의 30%)과 의도적 목표 주도 글쓰기(전문 지식 근로자의 70%) 모두를 지원합니다.

## 핵심 원칙: 모든 모드에서 중심이 살아나고 성장하는 방법

**근본적인 통찰**: 씨앗으로 시작하든 목표로 시작하든, 글쓰기는 **중심이 나타나고, 진화하며, 강화**될 때만 "생명"을 얻습니다.

### 중심이란 무엇인가?

**중심**은 글을 함께 묶는 구조적 축입니다. 중심은:
- **단순한 주제가 아님**: 주제는 "피드백 루프"(정적). 중심은 "시간 지연이 오류 전파를 복합화"(동적, 구조적)
- **살아있는 요소**: 약하게 시작하여 글을 쓰면서 강화됨
- **관계적**: 중심은 다른 요소(씨앗, 섹션, 논증)와의 관계 속에 존재
- **측정 가능**: 구조적 일관성을 나타내는 전체성 점수(1-10)를 통해

### 세 가지 모드에서의 중심 진화

#### 1. 씨앗 기반 모드: 연결에서 중심이 나타남

**프로세스**:
```
흩어진 씨앗 → AI가 잠재적 중심 발견 → 가장 강한 것 선택 → 글쓰기 → 중심 강화
```

**예시**:
- **씨앗**: Bill Evans 인용, 기타 연습 노트, 코드 리뷰 통찰, 나무 사진
- **발견된 중심** (약함): "연습과 숙달"
- **글쓰기 반복 1 후** (중간): "기술 개발에서 완전함 vs 근사"
- **반복 2 후** (강함): "작은 진실한 단계가 막연한 근사보다 더 빠르게 숙달로 복합화"
- **전체성 진행**: 6.5 → 7.8 → 9.1

**핵심**: 중심이 미리 정해지지 않았음 - 씨앗 관계에 대해 쓰면서 나타났고 수정을 통해 명확해짐.

---

#### 2. 목표 중심 모드: 마이크로 중심이 전체를 제공

**프로세스**:
```
목표 정의 → AI가 구조 제안 → 섹션 1 작성 → 섹션의 중심 발견 → 반복 → 통합
```

**예시**:
- **목표**: "4분기 프로젝트 회고록: 성과, 과제, 교훈"
- **섹션 1 중심** ("성과" 작성 중 나타남): "선행 지표로서의 사용자 유지율 급증"
- **섹션 2 중심** ("과제" 작성 중 나타남): "기술 부채가 기능 속도를 가림"
- **섹션 3 중심** ("교훈" 작성 중 나타남): "지표가 서사가 감추는 진실을 드러냄"
- **문서 수준 중심** (섹션 통합에서 나타남): "데이터 주도 회고가 반복된 실수를 방지"
- **전체성 진행**: 섹션은 각각 7.0으로 시작, 통합 문서는 8.5 도달

**핵심**: 미리 정해진 구조(목표)에서도 각 섹션이 작은 반복 단계를 통해 자체 살아있는 중심을 개발. 최종 문서의 전체성은 섹션-중심이 목표-중심을 얼마나 잘 지원하는지에 달려 있음.

---

#### 3. 하이브리드 모드: 중심이 목적과 통찰을 정렬

**프로세스**:
```
목표 정의 + 씨앗 수집 → AI가 목표 정렬 중심 발견 → 선택 → 글쓰기 → 중심이 양방향으로 심화
```

**예시**:
- **목표**: "피드백 루프의 시간 지연에 대한 문헌 리뷰"
- **씨앗**: 학술 논문 18개 + 개인 관찰 5개
- **발견된 하이브리드 중심**: "시간 지연 문제가 도메인 전반에 걸쳐 다르게 나타나지만 구조적 원인을 공유"
- **글쓰기 반복 1** (정렬: 0.85, 깊이: 0.70): 중심이 논문을 제공하지만, 통찰이 얕게 느껴짐
- **반복 2** (정렬: 0.92, 깊이: 0.85): 씨앗에서 교차 도메인 패턴 추가, 논문 초점을 유지하면서 이해 심화
- **최종** (정렬: 0.94, 깊이: 0.92): 중심이 논문을 증명하면서 예상치 못한 통찰(지연이 비선형적으로 복합화)을 드러냄
- **전체성 공식**: 정렬 × 깊이 = 0.94 × 0.92 = 0.86 (8.6/10)

**핵심**: 하이브리드 중심은 이중 제약을 만족해야 함: 목표를 제공(정렬)하면서 씨앗에서 진정한 통찰을 드러냄(깊이). 전체성이 두 차원을 모두 측정.

---

### 전체성: 중심 생명의 척도

**전체성**은 중심이 구조에 "생명"을 얼마나 잘 부여하는지를 정량화합니다:

**보편적 전체성 기준** (모든 모드):
1. **통일성**: 중심이 흩어진 요소를 함께 묶는가?
2. **명확성**: 중심의 핵심 아이디어가 날카롭고 구체적인가?
3. **강도**: 이 중심에서 글쓰기가 생성적으로 느껴지는가 (강제되지 않는가)?
4. **진화**: 중심이 반복 개선을 통해 심화되는가?

**모드별 전체성**:

| 모드 | 전체성 측정 | 목표 점수 |
|------|----------|---------|
| **씨앗 기반** | 중심이 흩어진 씨앗을 일관된 서사로 얼마나 잘 통합하는지 | 7.5+/10 |
| **목표 중심** | 섹션 전체성(내부 일관성) + 문서 전체성(섹션이 목표 제공) | 8.0+/10 |
| **하이브리드** | 정렬 점수(목표 제공) × 깊이 점수(통찰 드러냄) | 8.5+/10 |

**지속적인 피드백 루프**:
- WriteAlive는 중요한 편집 후마다 전체성 확인
- 낮은 전체성 (< 6.0) → AI 제안: "중심이 너무 막연할 수 있습니다. 더 구체적으로 만들 수 있나요?"
- 중간 전체성 (6.0-7.5) → AI가 중심을 강화하는 확장 방향 제안
- 높은 전체성 (7.5+) → AI 확인: "이 중심이 잘 유지되고 있습니다. 다른 섹션과의 연결을 고려하세요."

### 왜 중요한가: 살아있는 구조 vs 정적 개요

**전통적인 개요 접근법** (정적):
```
1. 서론
2. 본론 1
3. 본론 2
4. 결론
```
- 구조가 미리 정해짐
- 중심 없음, 생명 없음
- 글쓰기가 기계적으로 느껴짐
- 전체성: N/A (구조가 진화하지 않음)

**살리고 접근법 - 세 가지 모드 모두** (살아있음):
```
시작 → 작은 진실한 단계 쓰기 → 떠오르는 중심 발견 → 수정을 통해 강화 → 전체성 확인 → 반복
```
- 구조가 나타남(씨앗 기반) 또는 섹션이 자체 중심 개발(목표 중심) 또는 중심이 이중 목표 정렬(하이브리드)
- 중심이 글쓰기를 통해 강도 획득
- 글쓰기가 생성적으로 느껴짐
- 전체성: 지속적으로 측정되고 개선됨 (6.0 → 7.5 → 9.0)

**보장**: 반복을 통해 전체성이 증가하면, 글쓰기가 구조적 생명을 얻고 있는 것입니다. 전체성이 정체되거나 감소하면, 중심이 약하거나 글쓰기가 방향을 잃은 것입니다.

## 성공 지표

### 주요 KPI (구조적 생명 향상)
1. **글쓰기 빈도**: 사용자가 3배 더 자주 글을 씁니다 (주당 글쓰기 세션으로 측정)
2. **낮은 에너지 시작 성공**: 80%의 글쓰기 세션이 사전 계획 없이 시작됩니다 (도구 열기에서 첫 문장까지 < 2분)
3. **중심 진화 깊이**: 평균 문서가 완성 전에 5회 이상의 중심 찾기 반복을 거칩니다
4. **완성률**: 시작된 문서의 60%가 사용자 정의 "완료" 상태에 도달합니다 (업계 평균 ~20% 대비)

### 보조 KPI (사용자 참여)
5. **리텐션**: 70% 30일 리텐션, 50% 90일 리텐션
6. **AI 상호작용 품질**: 사용자가 AI 제안 중심/개선의 60% 이상을 수락합니다
7. **소리내어 읽기 사용**: 완성된 문서의 40%가 소리내어 읽기 개선 기능을 사용합니다
8. **버전 비교**: 사용자가 문서당 평균 3개 이상의 버전을 비교합니다

### 품질 지표 (글쓰기 우수성)
9. **통합성 점수**: 문단 응집력과 중심 명확성을 측정하는 맞춤 지표 (목표: 7+/10)
10. **사용자 자체 평가**: 80%의 사용자가 2주 후 "글쓰기가 더 쉽게 느껴진다"고 보고합니다
11. **피드백 통합**: 사용자가 버전 추적을 통해 외부 피드백을 2배 빠르게 통합합니다

### 구체적인 전후 비교 (튜토리얼 데이터)

**전통적인 글쓰기 방법** (일반적인 결과):
- **계획 단계**: 30-60분 개요 작성
  - 서론: 연습의 중요성
  - 본론 1: Bill Evans 철학 (더 많은 조사 필요...)
  - 본론 2: 내 경험에의 적용 (어떻게 구조화할까?)
  - 결론: 미래 약속 (진부하게 느껴짐)
- **빈 페이지 응시**: 15-30분
- **글쓰기**: 계획에 지쳐 종종 포기
- **결과**: 0단어 작성, 높은 좌절, 개요는 미사용
- **총 투자 시간**: 출력 없이 45-90분

**살리고 글쓰기 방법** (튜토리얼 결과):
- **월요일-목요일**: 4개 씨앗 캡처 (각 1-2분) = ~6분 분산
  - 지하철: Bill Evans 인용 #seed
  - 점심: 기타 연습 통찰 #idea #💡
  - 사무실: 코드 리뷰 깨달음 #seed
  - 공원 산책: 나무 사진 + 노트 #seed #nature
- **금요일 저녁 글쓰기 세션**: 30분
  - 씨앗 모으기: 5분
  - 중심 찾기 (AI): 3분 → "완전성 vs 근사치" (⭐⭐⭐) 식별
  - 초기 문단 작성: 5분 (낮은 에너지, 단지 3문장)
  - 옵션 A 확장 (구체적 경험): 10분 → 통합성 7.2/10
  - 옵션 B 확장 (나무 은유): 5분 → 통합성 8.3/10
  - 소리내어 읽고 다듬기: 2분 → 통합성 9.1/10
- **결과**: 600-800단어 완성 에세이, 출판 가능한 품질
- **총 시간**: ~35-40분 (분산 씨앗 캡처 포함)

**주요 차이점**:
- **에너지 장벽**: 높음 (전통적) vs 낮음 (살리고) - 3문장이면 시작 가능
- **구조**: 사전 결정 (전통적) vs 창발 (살리고) - 중심이 글쓰기를 통해 발견됨
- **완성**: <20% (전통적) vs 70%+ (살리고) - 튜토리얼 사용자 데이터
- **시간 효율**: 90분 = 0단어 (전통적) vs 35분 = 800단어 (살리고)
- **품질**: 해당 없음 (전통적, 완성 안 됨) vs 9.1/10 통합성 (살리고)
- **모바일 통합**: 없음 (전통적) vs 핵심 (살리고) - 씨앗의 50%가 모바일에서 캡처
- **통합성 진행**: 해당 없음 (전통적) vs 측정 가능 (살리고) - 7.2 → 8.3 → 9.1 추적

**목표 지표**:
- **"계획 마비" 90% 감소** - 사용자가 도구를 연 후 5분 이내에 글쓰기 시작
- **완성률 3배 증가** - 60% vs 20% 업계 기준선
- **시간 효율 2배** - 600단어 에세이를 30-40분에 완성 vs 전통적 방법 2시간 이상

## 목표 사용자 페르소나

### 1. 학술 작가 (주요)
- **프로필**: 논문, 논문, 학술 기사를 작성하는 대학원생, 연구자, 교수
- **고충**:
  - 개요 마비 상태에 빠짐 - 전체 구조가 명확해질 때까지 시작할 수 없음
  - 전통적인 학술 산문을 쓸 때 추진력을 잃음
  - 문단 통일성과 명확한 논증 유지에 어려움
- **목표**: 사전 계획의 마찰 없이 명확하고 잘 구조화된 학술 글쓰기를 생산합니다
- **성공 시나리오**: 단일 연구 질문으로 시작하여 유기적으로 진화시켜 학위 논문 장을 완성합니다

### 2. 전문 커뮤니케이터 (보조)
- **프로필**: 문서, 보고서, 제안서를 작성하는 제품 관리자, 컨설턴트, 기술 작가
- **고충**:
  - 작업 간 높은 컨텍스트 전환 비용으로 글쓰기 세션이 드뭅니다
  - 기업 글쓰기가 진부하고 형식적으로 느껴집니다
  - 전통적인 다중 초안 프로세스를 위한 시간 부족
  - **모바일 고충**: 통근 중 아이디어가 떠오르지만 전체 컨텍스트로 캡처하기 어렵습니다
- **목표**: 여러 기기에서 분산된 시간 블록으로 명확하고 설득력 있는 문서를 효율적으로 작성합니다
- **성공 시나리오**: 통근 중 모바일로 씨앗 캡처 → 데스크톱에서 전체 사양으로 개발

### 3. 성찰적 학습자 (보조)
- **프로필**: 개인 지식 기반을 유지하는 지식 노동자, 평생 학습자 (Obsidian, Notion 사용자)
- **고충**:
  - 노트가 단편화된 채로 남음 - 일관된 이해로 진화하지 않음
  - "아직 충분히 알지 못한다"는 두려움이 글쓰기를 방해합니다
  - 점진적으로 생각을 발전시킬 구조가 부족합니다
  - **모바일 고충**: 휴대폰으로 기사를 읽지만 노트가 글쓰기 워크플로와 연결되지 않습니다
- **목표**: 여러 기기에서 낮은 마찰 글쓰기를 통해 분산된 생각을 발전된 통찰로 변환합니다
- **성공 시나리오**: 휴대폰에서 책 하이라이트 캡처 → 씨앗으로 태그 → 주말에 데스크톱에서 에세이 작성

## 실제 사용자 여정: 일상의 순간에서 완성된 글쓰기까지

### 아이디어 → 글쓰기의 자연스러운 흐름

WriteAlive는 전용 "글쓰기 시간"이 있는 책상에서만이 아니라, 일상 생활에서 아이디어가 실제로 나타나는 방식을 지원하도록 설계되었습니다.

#### 여정 예시: "평일 씨앗 수집 → 주말 글쓰기"

**월요일 오전 7:30** - 지하철 통근, 휴대폰으로 Bill Evans 동영상 시청
📝 Obsidian 모바일에서 빠른 캡처:
```markdown
# 2025-11-04-subway-note.md
Bill Evans 동영상 봄.
"전체를 막연하게 근사하지 마라"는 말이 충격적이었다.
나는 항상 전체 기타 곡을 대충 연주하려고 했다.
#seed #practice
```

**화요일 오후 12:30** - 점심시간, 어제 기타 연습을 되돌아봄
📝 모바일 음성 메모 (Obsidian에 자동 동기화):
```markdown
# 2025-11-05-lunch-idea.md
어제 기타를 연습할 때
처음 4마디만 완벽하게 연습했다
나머지는 쉽게 따라왔다
#idea #guitar #💡
```

**수요일 오후 2시** - 사무실에서 코드 리뷰 중 갑자기 깨달음
📝 데스크톱 빠른 노트:
```markdown
# 2025-11-06-meeting-thought.md
코드 리뷰 중 깨달음:
작은 함수 하나를 제대로 작성하는 것이
전체 구조를 대충 그리는 것보다 낫다
이것도 Bill Evans의 접근법이지 않을까?
#seed #programming
```

**목요일 저녁** - 공원 산책, 나무 성장 패턴 관찰
📝 모바일 사진 + 노트:
```markdown
# 2025-11-07-park-walk.md
![[tree-growth.jpg]]
나무는 줄기에서 가지로 자란다.
글쓰기도 같은 방식으로 작동해야 하지 않을까?
#seed #nature
```

**금요일 오후 7시** - 드디어 글쓸 시간이 생겼지만 어디서 시작해야 할지 불확실

1. **데스크톱에서 Obsidian 열기**
2. **Ctrl/Cmd + P** → `WriteAlive: 씨앗 모으기` 명령
3. **WriteAlive가 결과 표시**:
   ```markdown
   🌱 발견된 씨앗 (4개)

   ✅ 2025-11-04 - "전체를 막연하게 근사하지 마라" #seed #practice
   ✅ 2025-11-05 - "처음 4마디만 완벽하게" #idea #guitar #💡
   ✅ 2025-11-06 - "작은 함수 하나를 제대로" #seed #programming
   ✅ 2025-11-07 - "나무는 줄기에서 가지로 자란다" #seed #nature
      [사진: tree-growth.jpg]

   모두 선택? [전체] [사용자 지정] [날짜별 필터]
   ```
4. **사용자가 "전체" 선택** → 새 노트 `내 연습 방법.md` 자동 생성
5. **AI가 씨앗을 분석하고 중심 제안**:
   ```
   🎯 발견된 중심:

   1. [강한 중심 ⭐⭐⭐] "완전성 vs 근사치"
      Bill Evans 철학의 핵심
      기타, 프로그래밍, 자연 성장을 연결

   2. [중간 중심 ⭐⭐] "부분에서 전체로 성장"
      나무 은유 + 연습 패턴

   3. [약한 중심 ⭐] "영역 간 적용"
      다른 분야에서 같은 원칙

   💡 추천: 중심 1부터 시작하세요 - 가장 생명력이 있습니다
   ```
6. **사용자가 첫 문단 작성** (5분, 낮은 에너지):
   ```markdown
   Bill Evans는 말했다: "전체를 막연하게 근사하지 마라.
   작은 부분을 택해 그것에 대해 완전히 진실되라."
   나는 이를 듣고 내 기타 연습을 되돌아봤다.
   ```
7. **AI가 확장 방향 제안**:
   ```
   🌿 글쓰기가 자라고 싶어하는 방향:

   옵션 A: 구체적 경험 추가 (⭐⭐⭐ 추천)
     "4마디를 완벽하게 연습했을 때 무슨 일이 일어났나요?
      독자가 그 경험을 느끼게 하세요."

   옵션 B: 자연 은유 사용 (⭐⭐)
     "나무의 성장이 당신의 연습을 어떻게 반영하나요?
      사진을 보면서 전개하세요."

   옵션 C: 프로그래밍과 연결 (⭐)
     "코드 리뷰에서 같은 발견?
      그 순간이 어땠나요?"
   ```
8. **사용자가 옵션 A 선택** → 구체적 기타 이야기 작성 (10분)
9. **사용자가 옵션 B 호출** → 나무 은유 추가 (5분)
10. **통합성 분석**: `7.2 → 8.3 → 9.1/10` (진행 추적)
11. **소리내어 읽기** → AI가 어색한 표현 발견 → 빠른 다듬기 (2분)
12. **스냅샷 저장**: "내 연습 방법 - 최종 (통합성 9.1)"

**결과**: 4일에 걸친 4개 모바일 캡처 → 30분 집중 글쓰기 → 통합성 9.1/10의 완성 에세이

**시간 분석 (총: ~35분)**:
- 월요일-목요일: 4 × 1-2분 모바일 캡처 = ~6분 (분산)
- 금요일 글쓰기 세션: 30분 (수집 5분 + 작성 25분)
- vs 전통적 방법: 1시간 이상 계획, 글쓰기 시작 안 함

**시연된 주요 사용성 원칙**:
- ✅ **낮은 에너지 장벽**: 4번의 빠른 모바일 캡처 (각 1-2분), 계획 없음
- ✅ **어디서나 캡처**: 지하철, 점심, 사무실, 공원 - 모두 글쓰기 씨앗이 됨
- ✅ **여러 날 수집**: 아이디어를 한 세션에 강제하지 않고 일주일에 걸쳐 수집
- ✅ **사진 통합**: 나무 사진이 글쓰기의 일부가 됨 (시각적 씨앗)
- ✅ **이모지 태그 지원**: `#💡`이 `#seed`, `#idea`와 동등하게 인식됨
- ✅ **음성 입력**: 모바일 음성 메모 자동 동기화 및 태그됨
- ✅ **연결자로서의 AI**: 4개의 분산된 씨앗에서 "완전성 vs 근사치" 중심 발견
- ✅ **측정 가능한 진행**: 통합성 점수 진행 (7.2 → 8.3 → 9.1)이 구조적 개선 표시
- ✅ **생성적 발견**: 중심과 확장이 사전 결정이 아닌 과정을 통해 나타남
- ✅ **기기 간 워크플로**: 모바일 캡처 → 데스크톱 글쓰기 (원활한 동기화)
- ✅ **기존 도구와 작동**: 순수 Obsidian + 마크다운, 독점 형식 없음

## 단계별 살리고 글쓰기 워크플로

이 섹션은 튜토리얼에서 경험한 완전한 살리고 글쓰기 프로세스를 각 단계의 시간 추정과 함께 설명합니다.

### 단계 1: 씨앗 수집 (총 5-10분)

**여러 날 수집 패턴** (권장):
- **월요일-목요일**: 하루 1-2개 씨앗 캡처 (각 1-2분)
  - 모바일 캡처: 통근, 점심시간, 회의, 산책 중
  - 빠른 태그 사용: `#seed`, `#idea`, `#💡`, `#thought`
  - 관련 시 사진 포함 (시각적 씨앗)
  - 음성 메모 자동 전사 및 동기화
- **총 분산 시간**: 4일에 걸쳐 ~6-8분
- **인지 부하**: 최소 (정리 없이 캡처만)

**단일 세션 수집** (대안):
- **금요일 저녁**: `WriteAlive: 씨앗 모으기` 명령 실행
  - 시스템이 볼트에서 태그된 모든 씨앗 찾기
  - 날짜 범위별 필터 ("이번 주", "이번 달")
  - 관련 씨앗 검토 및 선택
- **시간**: 5분
- **결과**: 글쓰기 준비된 4-10개 씨앗

### 단계 2: 중심 발견 (3-5분)

**명령**: `WriteAlive: 중심 찾기` (씨앗 문서 생성 후 자동 트리거)

**AI 분석**:
- 공통 주제에 대한 선택된 씨앗 분석
- 2-4개 잠재적 중심 식별
- 강도별 순위: 강함 (⭐⭐⭐), 중간 (⭐⭐), 약함 (⭐)
- 추천 제공: "중심 X부터 시작하세요 - 가장 생명력이 있습니다"

**예시 출력**:
```
🎯 발견된 중심:

1. [강한 중심 ⭐⭐⭐] "완전성 vs 근사치"
   모든 씨앗을 연결하는 핵심 개념

2. [중간 중심 ⭐⭐] "부분에서 전체로 성장"
   자연스러운 확장 패턴

3. [약한 중심 ⭐] "영역 간 적용"
   다른 맥락에서 같은 원칙
```

**사용자 행동**:
- 제안된 중심 검토
- 시작 중심 선택 (보통 가장 강한 것)
- 첫 문단 작성 시작

### 단계 3: 초기 글쓰기 (5-10분)

**낮은 에너지 시작**:
- 선택한 중심을 탐구하는 1-3문단 작성
- 완전한 구조나 개요 불필요
- 작은 부분에 대해 "완전히 진실"하기에 집중 (Bill Evans 원칙)
- AI는 관찰하지만 방해하지 않음

**예시 첫 문단** (튜토리얼에서):
```markdown
Bill Evans는 말했다: "전체를 막연하게 근사하지 마라.
작은 부분을 택해 그것에 대해 완전히 진실되라."
나는 이를 듣고 내 기타 연습을 되돌아봤다.
```

**결과**: 그 자체로 진정하고 완전하게 느껴지는 3-5문장

### 단계 4: 생성적 확장 (10-20분)

**명령**: `WriteAlive: 확장 제안`

**AI가 여러 방향 제공**:
```
🌿 글쓰기가 자라고 싶어하는 방향:

옵션 A: 구체적 경험 추가 (⭐⭐⭐ 추천)
  "4마디를 완벽하게 연습했을 때 무슨 일이 일어났나요?
   독자가 그 경험을 느끼게 하세요."

옵션 B: 자연 은유 사용 (⭐⭐)
  "나무의 성장이 당신의 연습을 어떻게 반영하나요?
   사진을 보면서 전개하세요."

옵션 C: 프로그래밍과 연결 (⭐)
  "코드 리뷰에서 같은 발견?
   그 순간이 어땠나요?"
```

**반복적 확장 프로세스**:
1. 사용자가 옵션 A 선택 → 구체적 경험 작성 (10분)
2. 통합성 확인: 7.2/10
3. 사용자가 옵션 B 선택 → 나무 은유 추가 (5분)
4. 통합성 확인: 8.3/10
5. 선택사항: 추가 깊이를 위해 옵션 C 선택

**핵심 원칙**: 각 확장이 이전 작업을 기반으로 하며 병렬 분기가 아님

### 단계 5: 통합성 분석 (2-3분)

**명령**: `WriteAlive: 통합성 분석`

**AI 구조적 피드백**:

```
✅ 강점:

  • 자연스러운 흐름: "듣기 → 반성 → 적용 → 이해"
  • 나무 은유가 연습 원칙 강화

⚠️ 개선 여지:
  • 프로그래밍 예시 아직 연결 안 됨 (미사용 씨앗)
  • 문단 2-3 사이 전환이 약간 갑작스러움
  • 결론이 서론으로 다시 연결될 수 있음

💡 다음 반복:
  옵션 C (프로그래밍) 추가하면 통합성이 9.0+ 도달 가능
  전환을 부드럽게 하면 흐름 개선
```

**진행 추적**:
- 초기: 7.2/10
- 1차 확장 후: 8.3/10
- 다듬기 후: 9.1/10
- **사용자가 구체적 개선 지표 확인**

### 단계 6: 다듬기 및 광택 (2-5분)

**AI 식별**:

- 불명확한 전환

### 에픽 0: 일일 노트에서 씨앗 수집 (MVP - 최우선순위)
**우선순위**: P0 (필수 - 핵심 차별화 요소)

**근거**: 이것은 분산된 일상 관찰과 일관된 글쓰기 사이의 다리입니다. 이것이 없으면 사용자는 시작하기 전에 아이디어를 수동으로 정리해야 하며, 이는 "낮은 에너지 장벽" 약속을 무효화합니다.

#### US-0.1: 볼트 전체 씨앗 수집
**역할**: 하루 종일 일시적인 아이디어를 캡처하는 지식 노동자
**원하는 것**: 시스템이 Obsidian 볼트 전체에서 태그된 씨앗을 자동으로 찾기
**이유**: 수백 개의 노트를 수동으로 검색하지 않고 글쓰기를 시작할 수 있도록

**수락 기준**:
- 사용자가 명령 호출: "씨앗 모으기" (Ctrl+P → WriteAlive: 씨앗 모으기)
- 시스템이 구성 가능한 씨앗 태그를 볼트에서 검색 (설정):
  - 기본값: `#seed`, `#writealive-seed`
  - 사용자가 맞춤 태그 추가 가능: `#idea`, `#thought`, `#fleeting`, `#💡` 등
  - 여러 태그 지원 (OR 로직): 구성된 태그 중 하나와 일치
  - 태그 별칭: `#씨앗` (한국어) = `#seed`
- 메타데이터와 함께 결과 반환 (모달/사이드바에):
  - 원본 노트 제목 및 경로
  - 씨앗 텍스트 (태그가 포함된 문단)
  - 생성 날짜
  - 관련 백링크 (있는 경우)
  - 사진 첨부 (있는 경우) 썸네일 미리보기와 함께
  - 일치된 태그 (예: `#seed`, `#💡`)
- 표시 형식 (튜토리얼에서):
  ```markdown
  🌱 발견된 씨앗 (4개)

  ✅ 2025-11-04 - "전체를 막연하게 근사하지 마라" #seed #practice
  ✅ 2025-11-05 - "처음 4마디만 완벽하게" #idea #guitar #💡
  ✅ 2025-11-06 - "작은 함수 하나를 제대로" #seed #programming
  ✅ 2025-11-07 - "나무는 줄기에서 가지로 자란다" #seed #nature
     [사진: tree-growth.jpg] 📷

  모두 선택? [전체] [사용자 지정] [날짜별 필터]
  ```
- 필터 옵션: "이번 주", "이번 달", "전체", "특정 폴더에서"
- 결과 정렬: 최신순 (기본값), 관련성, 또는 수동
- 사진 씨앗 지원:
  - 씨앗 텍스트와 함께 사진 썸네일 인라인 표시
  - 사진이 있는 씨앗 옆에 사진 아이콘 표시 (📷)
  - 클릭하여 전체 이미지 보기
  - 원본 노트에 `![[image.jpg]]`로 임베드된 사진

**실제 시나리오 (사진 지원 강화)**:
```
사용자의 맞춤 씨앗 태그 구성:
```

8. 사진이 에세이의 시각적 씨앗이 됨 (에세이에서 참조 가능)

**구조적 품질 지표**: 80%의 사용자가 첫 시도에서 씨앗을 성공적으로 모음; 수집 세션당 평균 5-10개 씨앗; 씨앗의 30%가 사진 첨부 포함

**수락 기준**:

**수락 기준**:
- 씨앗을 수집한 후 사용자가 "클러스터 제안" 호출 가능
- AI가 씨앗 내용을 분석하고 2-4개의 주제별 그룹 제안
- 각 클러스터 표시:
  - 제안된 주제 이름 (예: "실천 철학", "자연 성장 패턴")
  - 해당 클러스터의 씨앗
  - 설명: "이 씨앗들은 다음에 초점을 공유합니다..."
- 사용자가 클러스터를 수락, 병합, 분할 또는 무시 가능
- "클러스터 [X]에서 글쓰기 시작"이 해당 씨앗만으로 문서 생성

**구조적 품질 지표**: 클러스터된 씨앗이 30% 더 일관성 있는 첫 번째 초안으로 이어짐 (통합성 점수로 측정)

---

#### US-0.4: MOC (내용 지도) 통합
**역할**: MOC 구조를 사용하여 노트를 정리하는 지식 노동자
**원하는 것**: WriteAlive가 내 MOC를 인식하고 글쓰기 시작점으로 사용하기
**이유**: 글쓰기를 위해 기존 지식 조직을 활용할 수 있도록

**수락 기준**:
- 사용자가 플러그인 설정에서 노트를 MOC로 지정 가능 (폴더 경로, 태그 `#moc`, 또는 YAML `type: moc`를 통해)
- "MOC에서 시작" 명령이 사용 가능한 MOC 목록 표시
- MOC 미리보기 표시:
  - 제목 및 구조
  - 링크된 노트 수
  - 마지막 수정 날짜
  - 링크의 빠른 미리보기
- MOC 선택 시 다음과 함께 새 문서 생성:
  - 모든 링크된 노트를 컨텍스트로 가져옴
  - 계층 구조 보존 (H2/H3 섹션 → 문서 섹션)
  - 전체 노트 내용 또는 발췌만 포함 옵션
- 자동 동기화: MOC가 업데이트되면 WriteAlive가 글쓰기 문서 새로 고침 제안

**실제 시나리오**:
```markdown
<!-- 사용자의 MOC: "창의성과 실천.md" -->
# 창의성과 실천

## 기초 개념
- [[Christopher Alexander - 중심]]
- [[Bill Evans - 근사치보다 진실]]
- [[자연 성장 패턴]]

## 개인 관찰
- [[2025-10-28 - Nature of Order 읽기]]
- [[2025-10-29 - 산책 관찰]]

## 응용
- [[살리고 글쓰기 방법]]
- [[글쓰기에서의 의도적 연습]]

---
WriteAlive 명령: "MOC에서 시작"
→ "창의성과 실천" 선택
→ 모든 7개 링크된 노트를 초기 컨텍스트로 새 문서 생성
→ AI 제안: "MOC에 3개 주제가 있습니다. 먼저 어떤 것을 탐색하시겠습니까?"
```

**구조적 품질 지표**:
- MOC 시작 문서의 완성률이 씨앗 시작보다 50% 높음 (85% 대 70%)
- MOC당 평균 10-15개 링크된 노트
- MOC 워크플로를 가진 사용자의 60%가 이 기능 채택

---

#### US-0.5: 자동 MOC 생성 (선택 사항, MVP 이후)
**역할**: 관련 노트를 축적하지만 수동으로 MOC를 생성하지 않는 작가
**원하는 것**: WriteAlive가 노트 관계를 기반으로 MOC 후보를 제안하기
**이유**: 새로운 지식 구조를 발견할 수 있도록

**수락 기준**:
- "MOC 제안" 명령이 볼트를 분석:
  - 5개 이상의 외부 링크가 있는 노트
  - 상호 백링크가 있는 노트 클러스터
  - 공통 태그/폴더를 공유하는 노트
- MOC 후보를 다음과 함께 제시:
  - 내용 기반 제안 제목
  - 제안된 구조 (계층적 그룹화)
  - 강도 점수 (클러스터의 응집력)
- 사용자가 수락, 편집 또는 거부 가능
- 원클릭 "MOC 생성 + 글쓰기 시작"이 MOC 노트와 글쓰기 문서 모두 생성

**구조적 품질 지표**: 제안된 MOC의 40%가 1주일 이내에 수락되고 글쓰기에 사용됨

---

#### US-0.6: 살아있는 MOC 자동 업데이트
**역할**: 매일 씨앗을 캡처하는 지식 노동자
**원하는 것**: MOC가 관련 태그가 있는 새 씨앗을 자동으로 반영하기
**이유**: 수동 유지 관리 없이 지식 지도를 최신 상태로 유지

**수락 기준**:
- 사용자가 MOC 프론트매터에서 "살아있는 MOC" 모드 활성화:
  ```yaml
  ---
  writealive:
    auto_gather_seeds: true
    seed_tags: [creativity, practice, writing, 창의성]  # 여러 태그 지원
    seed_tag_mode: any  # 'any' (OR) 또는 'all' (AND)
    update_frequency: daily  # 또는: realtime, manual
    include_global_seed_tags: true  # 글로벌 설정의 태그도 사용
  ---
  ```
- MOC에 자동 업데이트 섹션 마커 포함:
  ```markdown
  ## 최근 씨앗 (자동 업데이트)
  <!-- BEGIN WRITEALIVE-AUTO -->
  <!-- END WRITEALIVE-AUTO -->
  ```
- WriteAlive가 MOC의 `seed_tags`와 일치하는 새 노트를 볼트에서 모니터링
- 일치하는 새 씨앗 발견 시:
  - **실시간 모드**: AUTO 섹션에 즉시 추가
  - **일일 모드**: 알림 표시: "'창의성' MOC에 3개 새 씨앗" [지금 업데이트] [검토]
  - **수동 모드**: "MOC에서 시작" 명령 중 제안
- 자동 섹션 정렬: 최신순 (기본값), 관련성, 또는 맞춤
- 사용자가 AUTO에서 수동 섹션으로 항목 이동 가능 (영구화)
- 사용자 편집 존중: 수동 콘텐츠를 절대 덮어쓰지 않음

**실제 시나리오**:
```markdown
사용자의 MOC: "창의성과 실천.md"
---
writealive:
  auto_gather_seeds: true
  seed_tags: [creativity, practice, 창의성, 연습]  # 다국어!
  seed_tag_mode: any
  update_frequency: daily
---

월요일: 책 읽기 → "Alexander: 중심" #creativity #씨앗
→ MOC의 "creativity" 태그와 일치
→ [알림] "'창의성과 실천' MOC에 1개 새 씨앗"

화요일: 비디오 시청 → "Evans: 진실" #practice #idea
→ MOC의 "practice" 태그 + 글로벌 "#idea" 태그와 일치
→ 자동 섹션에 이제 2개 씨앗

수요일: 한국어 노트 → "자연스러운 성장" #창의성 #씨앗
→ MOC의 "창의성" 태그 (한국어)와 일치
→ 자동 섹션에 이제 3개 씨앗

일요일: "MOC에서 시작"
→ 모든 3개 자동 수집 씨앗 + 수동 링크를 컨텍스트로
→ AI: "이번 주 MOC가 성장했습니다. 이중언어 주제를 발견했습니다..."
```

**세 가지 구현 접근 방식** (MVP용으로 1개 선택):

1. **스마트 마커** (MVP 권장)
   - 장점: 순수 마크다운과 작동, 사용자 제어, 명확한 경계
   - 단점: 마커 규율 필요

2. **Dataview 통합** (Dataview 플러그인 사용 가능한 경우)
   - 장점: 실시간, 수동 업데이트 불필요
   - 단점: Dataview 플러그인 필요, 이식성 낮음

3. **하이브리드 알림** (가장 안전)
   - 장점: 사용자가 항상 제어, 우발적 덮어쓰기 없음
   - 단점: 사용자 작업 필요, 진정한 "자동" 아님

**구조적 품질 지표**:
- MOC 사용자의 50%가 자동 업데이트 활성화
- 자동 업데이트 MOC가 수동 MOC보다 2-3배 더 많은 씨앗 수신
- 자동 수집 씨앗의 70%가 유지됨 (무관하다고 제거되지 않음)
- 절약된 시간: 주당 5-10분의 수동 MOC 유지 관리

**엣지 케이스 및 안전**:
- AUTO 마커 외부의 콘텐츠를 절대 수정하지 않음
- 마커가 없으면 추가 제안 (자동 주입하지 않음)
- 여러 MOC가 동일한 씨앗 태그와 일치하면 업데이트할 MOC를 사용자에게 묻기
- 실행 취소 지원: "마지막 자동 업데이트 되돌리기"

### 에픽 1: 낮은 에너지 글쓰기 시작 (MVP)
**우선순위**: P0 (필수)
**노력**: 대형 (8-13 스토리 포인트)
**근거**: 근본적인 차별화 요소 - 글쓰기를 시작하는 에너지 장벽 제거

#### US-1.1: 씨앗 기반 시작
**역할**: 막연한 아이디어를 가진 작가
**원하는 것**: 단일 문장이나 구절로만 글쓰기 시작하기
**이유**: 완전한 개요나 구조 없이 시작할 수 있도록

**수락 기준**:
- 사용자가 단일 문장을 씨앗으로 새 문서 생성 가능
- 도구가 모든 길이의 텍스트를 즉시 수락 (3단어에서 3문단까지)
- 필수 필드, 템플릿, 구조적 프롬프트 없음
- 사용자에게 격려 표시: "시작하기에 충분합니다. 여기서 어떤 중심을 보시나요?"
- 첫 번째 저장이 흐림에서 자동으로 발생 (저장 마찰 없음)

**구조적 품질 지표**: 첫 문장까지 에너지 < 2분, 90% 성공률

---

#### US-1.2: 언제 어디서나 글쓰기
**역할**: 분산된 시간을 가진 바쁜 전문가
**원하는 것**: 여러 기기와 세션에서 글쓰기를 원활하게 계속하기
**이유**: 통근, 점심 시간, 또는 영감이 떠오를 때마다 글을 쓸 수 있도록

**수락 기준**:
- 마크다운 파일이 GitHub를 통해 동기화 (사용자가 저장소 제공)
- 모바일 반응형 인터페이스 (또는 모바일용 Obsidian 플러그인)
- 충돌 해결과 함께 30초마다 자동 저장
- 세션 재개가 표시: "[시간 전]에 마지막 작업. 현재 중심: [X]"
- 동기화 대기열이 있는 오프라인 모드

**구조적 품질 지표**: 40%의 세션이 모바일 기기에서 발생

### 에픽 2: AI 지원 중심 발견 (MVP)
**우선순위**: P0 (필수)
**노력**: 대형 (13-21 스토리 포인트)
**근거**: 살리고 글쓰기를 비전문가도 접근 가능하게 만드는 핵심 AI 기능

#### US-2.1: 중심 식별
**역할**: 초안을 개발하는 작가
**원하는 것**: AI가 기존 텍스트에서 잠재적 중심을 제안하기
**이유**: Alexander의 방법론에 대한 전문 지식 없이 확장을 위한 구조적 축을 식별할 수 있도록

**수락 기준**:
- 사용자가 문단이나 섹션을 선택하고 "중심 찾기" 명령 호출
- AI가 GPT-4/Claude를 사용하여 텍스트를 분석하고 설명과 함께 2-4개의 잠재적 중심 반환
- 각 중심 표시: "이 구절이 에너지를 보유하는 것 같습니다 왜냐하면..."
- 사용자가 제안을 수락, 수정 또는 거부 가능
- 수락된 중심이 시각적으로 강조 표시됨 (미묘한 사이드바 표기)
- AI 고려 사항: 반복되는 개념, 감정적으로 충전된 언어, 구조적 축, 미해결 긴장

**구조적 품질 지표**: 사용자가 제안된 중심의 50% 이상을 수락

---

#### US-2.2: 생성적 확장 프롬프트
**역할**: 씨앗 아이디어를 응시하는 작가
**원하는 것**: AI가 현재 중심에 대한 확장 전략을 제안하기
**이유**: 작가의 블록을 극복하고 여러 방향을 탐색할 수 있도록

**수락 기준**:
- 사용자가 선택된 텍스트에서 "이 중심 키우기" 호출
- AI가 3-5개의 생성적 질문/프롬프트 제안:
  - "이것을 설명하는 구체적인 예는 무엇인가요?"
  - "반대 관점은 무엇인가요?"
  - "이것을 시간상 거슬러 추적하면 어떻게 되나요?"
  - "이 관찰의 기초가 되는 구조는 무엇인가요?"
- 사용자가 하나의 프롬프트를 선택하면 AI가 2-3개의 시작 문장 생성
- 사용자가 반복 가능: "더 많은 방향 보기"
- 모든 제안이 비침입적 (모달/사이드바, 쉽게 무시 가능)

**구조적 품질 지표**: 막힌 세션의 70% (>5분 텍스트 없음)가 프롬프트 사용 후 성공적으로 재개됨

---

#### US-2.3: 통합성 분석
**역할**: 여러 문단을 가진 작가
**원하는 것**: AI가 문서의 통합성/일관성을 평가하기
**이유**: 약한 연결, 중복 또는 누락된 부분을 식별할 수 있도록

**수락 기준**:
- 사용자가 전체 문서 또는 섹션에서 "통합성 분석" 호출
- AI 평가:
  - 문단 통일성 (각 문단이 하나의 명확한 메시지를 가지고 있나요?)
  - 문단 간 전환 (아이디어가 자연스럽게 흐르나요?)
  - 중심 계층 (주요 아이디어가 명확하게 지배적인가요?)
  - 구조적 간격 (생각을 완성하기 위해 무엇이 누락되었나요?)
- 출력: 강한 연결 (녹색), 약한 링크 (노란색), 간격 (빨간색)을 표시하는 시각적 지도
- 구체적 제안: "문단 3과 7이 같은 아이디어를 반복하는 것 같습니다 - 병합 고려"
- 통합성 점수: 설명과 함께 1-10

**구조적 품질 지표**: 7+ 통합성 점수를 가진 문서가 사용자 평가에 따라 90% 완성됨

### 에픽 3: 반복적 개선 도구 (MVP)
**우선순위**: P0 (필수)
**노력**: 중간 (5-8 스토리 포인트)
**근거**: 살리고 글쓰기의 반복적 특성 지원

#### US-3.1: 소리내어 읽기 검토
**역할**: 초안을 완성하는 작가
**원하는 것**: AI 음성으로 텍스트를 듣기
**이유**: 어색한 표현, 리듬 문제, 불명확한 구절을 식별할 수 있도록

**수락 기준**:
- 사용자가 문단 또는 전체 문서에서 "소리내어 읽기" 버튼 클릭
- 자연스러운 억양의 텍스트 음성 변환 (브라우저 API 또는 서비스)
- 읽기가 현재 문장을 강조 표시
- 일시정지/재개 제어
- 사용자가 듣는 동안 클릭하여 편집 가능 (재생 일시정지)
- 속도 제어: 0.8x, 1x, 1.2x

**구조적 품질 지표**: 문서의 40%가 "완료"로 표시하기 전에 소리내어 읽기 사용

#### US-3.2: 버전 스냅샷 및 비교
**역할**: 반복을 통해 문서를 진화시키는 작가
**원하는 것**: 명명된 스냅샷을 저장하고 버전을 비교하기
**이유**: 구조적 진화를 보고 잠재적으로 실험을 되돌릴 수 있도록

**수락 기준**:
- 사용자가 "스냅샷 저장"을 클릭하고 이름 지정 (예: "중심 3 확장 후")
- 스냅샷 저장: 전체 텍스트, 타임스탬프, 현재 중심, 통합성 점수
- "버전 기록"이 스냅샷의 타임라인 표시
- 사용자가 두 버전을 선택하여 나란히 diff 보기
- Diff 강조 표시: 추가 (녹색), 삭제 (빨간색), 이동된 텍스트 (파란색)
- 모든 스냅샷으로 원클릭 되돌리기

**구조적 품질 지표**: 사용자가 문서당 평균 5개 스냅샷 생성; 20%가 비교 수행

### 에픽 4: 학술 글쓰기 구조 지원 (MVP)
**우선순위**: P0 (필수)
**노력**: 소형 (3-5 스토리 포인트)
**근거**: 생성적 프로세스를 지원하면서 익숙한 학술 규칙으로의 다리

#### US-4.1: 문단 라벨링
**역할**: 학술 작가
**원하는 것**: 선택적으로 문단에 수사적 목적 라벨 지정하기
**이유**: 경직된 개요 없이 구조적 명확성을 유지할 수 있도록

**수락 기준**:
- 사용자가 라벨로 모든 문단에 태그 지정 가능: [주장], [증거], [분석], [반론], [전환], [예시], [맥락], [맞춤]
- 라벨이 미묘한 여백 주석으로 표시됨 (인라인 아님)
- "구조 보기"가 문서를 라벨된 개요로 표시
- AI가 라벨 제안 가능: "이 문단은 문단 2의 주장에 대한 증거를 제공하는 것 같습니다"
- 라벨은 선택 사항 - 진행하는 데 절대 필요하지 않음

**구조적 품질 지표**: 학술 사용자의 60%가 문단 라벨링 채택

#### US-4.2: 섹션 일관성 확인
**역할**: 학술 작가
**원하는 것**: 각 문단이 통일성을 가지고 있는지 확인하기 (하나의 명확한 메시지)
**이유**: 글쓰기가 문단 구조에 대한 학술 표준을 충족하도록

**수락 기준**:
- 사용자가 모든 문단에서 "문단 통일성 확인" 호출
- AI 평가: 이 문단이 하나의 주요 아이디어를 가지고 있나요?
- 단편화된 경우 제안: "이 문단은 [A]와 [B]를 논의합니다 - 분할 고려"
- 통일된 경우 확인: "[주요 아이디어]에 대한 명확한 통일성"
- 전체 문서에 대한 일괄 확인 가능

**구조적 품질 지표**: 통일성 확인을 통과한 문서가 고문/검토자의 수정 요청이 40% 적음

### 에픽 6.5: 모바일 우선 기능 (MVP - 높은 우선순위)
**우선순위**: P0 (모바일 사용자에게 필수)

#### US-6.5.1: 모바일 최적화 씨앗 캡처
**역할**: 걷거나 통근하는 모바일 사용자
**원하는 것**: 최소한의 마찰로 씨앗 아이디어를 빠르게 캡처하기

**수락 기준**:

  - 오프라인 작동 (기기 STT)
- **최소 키보드**: 최근 사용에 기반한 태그 자동 제안
- **자동 저장**: 2초마다 저장 (모바일 네트워크 불안정)
- **오프라인 모드**: 씨앗을 로컬에서 대기열에 넣고 온라인 시 동기화
- **사진 씨앗**: 사진 + 빠른 노트 캡처 → 씨앗으로 태그 지정
  - 예시: 책 페이지 사진 + "Alexander의 중심 개념" #seed #reading

**모바일 특정 제약**:
- Android에서 작동 (Obsidian Mobile 1.4.0+)
- iOS에서 작동 (Obsidian Mobile 1.4.0+)
- 복잡한 UI 없음 (제한된 화면 공간)
- 터치 최적화 (호버 상태 없음)
- 성능: 중급 기기에서 <100ms UI 응답

**실제 시나리오**:
```
오전 7:30 - 지하철에서
사용자가 책에서 영감을 주는 인용문 발견
→ Obsidian Mobile 열기
→ "빠른 씨앗" 버튼 탭
→ 페이지 사진 촬영
→ 말하기: "Alexander가 강한 중심이 약한 중심을 더 강하게 만든다고 말함"
→ 자동 태그: #seed #reading #christopher-alexander
→ 5초 이내 저장
→ 읽기로 돌아가기

저녁 - 책상에서
→ 데스크톱에서 "씨앗 모으기" 열기
→ 사진과 함께 아침 지하철 씨앗 찾기
→ 에세이 쓰기 시작
```

**구조적 품질 지표**: 모바일에서 캡처된 씨앗의 70%가 7일 이내에 데스크톱 글쓰기에 사용됨

- **씨앗 미리보기**: 씨앗 탭 → 모달에서 전체 노트 표시
- **태그별 필터**: 특정 주제만 볼 수 있는 빠른 필터
- **스와이프 탐색**: MOC 간 스와이프
- **MOC 북마크**: 자주 액세스하는 MOC를 맨 위에 고정

**모바일에서 지원되지 않음 (데스크톱 전용)**:

- 통합성 분석 (시각적 복잡성)

**사용 사례**:
```

→ MOC "창의성과 실천" 열기
→ 이번 주 5개 새로운 자동 수집 씨앗 보기

→ 전체 노트 읽기
→ 정신적 노트: "이것이 Alexander와 연결됩니다!"
→ 저녁 책상에서: 연결에 대해 쓰기
```

**구조적 품질 지표**: 모바일 사용자의 50%가 주간 MOC 찾아보기; 30%가 데스크톱 글쓰기 세션 전에 모바일에서 씨앗 검토

## 비기능적 요구사항

### 성능
- **지연 시간**: AI 응답 (중심 식별, 확장) < 90백분위수에 대해 3초
- **오프라인**: 전체 마크다운 편집이 오프라인에서 작동; AI 기능이 동기화를 위해 대기열에 추가됨
- **대용량 문서**: 성능 저하 없이 최대 50,000단어의 문서 지원
- **동시 세션**: CRDT 또는 OT로 2개 기기에서 동일한 문서를 편집하는 사용자 처리

### 보안 및 개인정보
- **API 키**: 로컬에서 암호화 저장 (브라우저 crypto API) 또는 사용자 제어 env 파일에
- **데이터 저장소**: 사용자가 데이터를 소유 - GitHub 저장소 또는 로컬 파일 시스템의 마크다운 파일
- **중앙 저장소 없음**: WriteAlive가 서버에 문서 내용을 절대 저장하지 않음 (상태 비저장 API 호출)
- **감사 추적**: 모든 AI 요청이 검토를 위해 타임스탬프와 함께 로컬에 기록됨
- **GDPR 준수**: 최소 데이터 수집; 사용자가 모든 데이터를 내보내기/삭제 가능

### 확장성
- **상태 비저장 백엔드**: 모든 AI 오케스트레이션 상태 비저장 (수평 확장 가능)
- **속도 제한**: API 비용 초과를 방지하기 위한 클라이언트 측 속도 제한 구현
- **캐싱**: 동일한 요청에 대한 AI 응답 캐싱 (24시간 TTL)으로 비용 절감
- **일괄 처리**: 여러 AI 요청을 대기열에 넣고 최적화된 일괄 처리로 처리

### 접근성
- **WCAG 2.1 AA**: 키보드 탐색, 스크린 리더 지원, 충분한 대비
- **소리내어 읽기**: 난독증 또는 시각 장애가 있는 사용자 지원
- **맞춤형 UI**: 글꼴 크기, 간격, 색상 테마 (고대비 모드)
- **모바일 터치**: 큰 터치 대상 (최소 44x44px), 버전 비교를 위한 스와이프 제스처

### 국제화
- **주요**: 한국어 (한국어) 및 영어 (미국)
- **UI 번역**: i18n 파일의 모든 인터페이스 문자열 (react-i18next 또는 유사)
- **LLM 언어 지원**: 문서 언어를 감지하고 그에 따라 AI 프롬프트
- **RTL 고려**: 향후 확장을 위한 RTL 언어를 지원하는 아키텍처

## 기술적 제약 및 결정

### 플랫폼 고려사항

**결정 포인트**: Obsidian 플러그인 vs 독립 실행형 앱 vs 둘 다?

**옵션 A: Obsidian 플러그인 우선** (MVP 권장)
- **장점**:
  - 지식 노동자의 기존 사용자 기반
  - 마크다운 네이티브 환경
  - Obsidian의 동기화, 모바일 및 플러그인 생태계 활용
  - 더 빠른 MVP (전체 편집기 구축 불필요)
- **단점**:
  - 초기에는 Obsidian 사용자로 제한
  - 플러그인 API 제약 (하지만 우리 필요에는 충분)
  - 커뮤니티 플러그인 승인 프로세스
- **노력**: 중간 (5-8주)

**옵션 B: 독립 실행형 Electron 앱**
- **장점**:
  - UX/UI에 대한 완전한 제어
  - Obsidian이 아닌 사용자에게 도달
  - 네이티브 OS 통합 가능
- **단점**:
  - 전체 마크다운 편집기 구축 필요 (높은 복잡성)
  - 더 느린 MVP (10-16주)
  - 크로스 플랫폼 테스트 부담
- **노력**: 대형 (12-20주)

**옵션 C: 웹 앱 (React + Monaco/CodeMirror)**
- **장점**:
  - 브라우저를 통해 액세스 가능 (설치 불필요)
  - 가장 쉬운 크로스 플랫폼 배포
  - 모바일용 프로그레시브 웹 앱
- **단점**:
  - Obsidian/Electron보다 덜 네이티브 느낌
  - 동기화 복잡성 (서비스 구축 또는 통합 필요)
  - 브라우저의 API 키 저장 (보안 트레이드오프)
- **노력**: 중-대형 (8-14주)

**권장 사항**: MVP를 위해 **옵션 A (Obsidian 플러그인)**로 시작하여 실제 사용자와 살리고 글쓰기 방법론을 빠르게 검증합니다. 더 넓은 도달을 위해 MVP 이후 옵션 C (웹 앱)로 확장합니다. 옵션 B (Electron)는 강력한 사용자 수요와 리소스가 허용하는 경우에만.

## 위험 및 완화 전략

### 위험 1: AI 응답 품질 불일치
**심각도**: 높음
**확률**: 중간
**영향**: 중심 제안이 자주 무관하면 사용자가 신뢰를 잃음

**완화**:
- 실제 살리고 글쓰기 문서의 예시를 포함한 광범위한 프롬프트 엔지니어링
- 출시 전 내부 사용자와 프롬프트 A/B 테스트
- 사용자 피드백 수집: "이 제안이 도움이 되었나요?" 엄지 위/아래로
- 빠르게 반복하기 위한 프롬프트 버전 관리 구현
- AI 신뢰도가 낮으면 더 간단한 휴리스틱으로 폴백
- 명확한 면책 조항: "AI 제안은 처방이 아니라 시작점입니다"

### 위험 2: 사용자의 API 비용 초과
**심각도**: 중간
**확률**: 중간
**영향**: 사용자가 높은 API 청구서에 놀라고, 부정적 리뷰

**완화**:
- 각 AI 호출 전 눈에 띄는 비용 추정기
- 50%, 80%, 100%에서 경고가 있는 월별 예산 설정
- 클라이언트 측 속도 제한 (시간당 최대 X 호출)
- 효율적인 프롬프트 (품질을 희생하지 않고 토큰 최소화)
- 동일한 요청을 24시간 동안 캐시
- 비용 비교 가이드 제공: "중심 찾기는 일반적으로 $0.01 비용"

- Bill Evans 원칙: "자신의 기초를 구축하세요, 도구는 보조합니다"

**영향**: 글쓰기가 손실되면 재앙적인 사용자 신뢰 상실

**완화**:

- 명확한 충돌 해결 UI (절대 무음 덮어쓰기 없음)

- 카나리아 테스트: 스테이징 환경에서 충돌 시뮬레이션
- 사고 대응 계획: 데이터 손실이 보고되면 우선순위 0

### 위험 7: 긴 문서에 대한 제한된 AI 컨텍스트 윈도우
**심각도**: 중간
**확률**: 높음
**영향**: AI 제안이 >10k 단어 문서에서 덜 관련성 있게 됨

**완화**:

- 사용자가 명시적으로 컨텍스트 제공 가능: "이 이전 섹션도 고려하세요"
- 긴 문서를 위한 점진적 요약 (각 섹션에서 주요 중심 추출)
- 기대치 설정: "20k 단어 미만 문서에서 가장 잘 작동"
- 미래: 전체 문서 컨텍스트를 위한 RAG (검색 증강 생성)

## 설계 원칙 (변환 중심)

### 1. 구조적 생명 보존
- 모든 기능이 응집력을 향상시켜야 하며 주의를 분산시키지 않음
- 모달 대화 상자 최소화 - AI 제안에 인라인/사이드바 사용
- 시각적 계층: 글쓰기 콘텐츠가 항상 주요, 도구가 보조
- 사용자의 몰입 상태 존중 - 중단하는 알림 없음

### 2. 반복보다 생성적 순서
- 개발을 "다음 작은 구조적 개선은 무엇인가?"로 프레임화
- 각 변환이 한 측면을 향상: 중심 발견, 통합성 분석, 개선
- "버전 2.0 재작성" 없음 - 점진적으로 기능 진화
- 영향 문서화: "이 변경이 문단 통일성 감지를 15% 향상"

### 3. 근사치보다 진실 (Bill Evans 원칙)
- 많은 것을 서투르게 하는 것보다 한 가지를 완벽하게 하는 것이 좋음
- MVP는 핵심 살리고 루프에 집중: 씨앗 → 중심 → 확장 → 개선
- "충분히 좋은" AI 프롬프트 없음 - 실제 글쓰기 샘플로 엄격하게 테스트
- 중요한 것을 측정: 완성률, 기능 수가 아님

### 4. 사용자와 공동 설계
- 사용자는 만족시킬 고객이 아니라 협력자
- 초기 베타 프로그램이 사용자를 공동 창작자로 취급
- MVP 개발 중 주간 피드백 세션
- 변경 로그에서 변환 결정을 투명하게 설명
- 사용자 시나리오가 직접 변환 의도가 됨

### 5. 살아있는 문서화
- 이 PRD는 제품과 함께 진화 - 시작 시 동결되지 않음
- 제공된 각 변환이 관련 섹션을 업데이트
- 미해결 질문이 해결되고 문서화됨
- 성공 지표가 분기별로 검토되고 학습에 따라 조정됨
